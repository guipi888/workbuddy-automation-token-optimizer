# 定时任务管理（含 token 节约）

定时任务全生命周期管理技能：审计 token 消耗 + 评估可合并任务 + 识别可删除任务，一次触发输出综合优化报告。

## 三大核心能力

### 能力 1：Token 消耗审计

自动读取数据库，统计上一个自然月每个定时任务的 token 消耗，找出高消耗任务。

### 能力 2：合并评估

扫描所有定时任务，按合并规则自动评估哪些任务可以合并为一个。

### 能力 3：冗余清理建议

识别长期暂停、功能被替代、重复执行的任务，给出删除建议。

---

## 合并评估规则

### 规则 1：时间相近 + 领域相同 → 高可合并性

同一领域、触发时间间隔 ≤ 30 分钟的任务，几乎总是可以合并。

| 示例 | 任务 A | 任务 B | 间隔 | 判定 |
|------|--------|--------|------|------|
| 周六 10:00 实盘复盘 | 周六 10:30 虚拟盘复盘 | 30min | ✅ 高 |
| 周日 02:00 归档上传 | 周日 任意 备份上传 | 任意 | ✅ 高（同天同域） |
| 周一 09:00 凭证周检 | 周一 09:00 内容资产挖掘 | 0min | ✅ 高（同时触发） |

### 规则 2：功能覆盖/替代 → 应删除旧任务

当旧任务的功能已被新任务完全覆盖时，标记为「可删除」。

| 场景 | 旧任务 | 新任务 | 判定 |
|------|--------|--------|------|
| 日报类被合并 | 每日资讯采集 | 每日资讯日报+部署+海外增强 | 🗑️ 删除 |
| 单功能被多功能替代 | 任务列表扫描 | 知识星球提问+任务扫描（合并版） | 🗑️ 删除 |

### 规则 3：长期 PAUSED → 建议删除

状态为 PAUSED 且超过 14 天未恢复的任务，标记为「可删除」。需逐条确认是否有保留价值。

### 规则 4：功能重复 → 保留一个

多个任务做同一件事，只保留最完善的版本。

### 规则 5：高频低价值 → 降频

每日执行但实际不需要每日做的任务，建议降频（如每2天或每周）。

| 指标 | 阈值 | 建议 |
|------|------|------|
| 每日执行 | 连续 7 天输出变化 < 5% | 降频为每2天 |
| 每小时执行 | 非实时监控场景 | 降频为每6小时 |

## 合并操作规范

### 合并后的任务结构原则

合并任务必须遵循「一次触发，多件事，一份报告」原则：

```
你是「{领域}」多任务合并机器人。每次触发执行以下 N 件事，最后输出一份统一结论报告。

---

## 任务 1：{原任务A名称}
{原任务A的关键步骤}

---

## 任务 2：{原任务B名称}
{原任务B的关键步骤}

---

## 报告格式

📊 {领域}报告 — {今天 YYYY-MM-DD}

{各任务结论汇总}
```

### 合并操作步骤

1. 读取原任务 A 的 prompt（确认完整功能）
2. 读取原任务 B 的 prompt（确认完整功能）
3. 合并 prompt → 更新保留的任务
4. 删除被吞并的旧任务
5. 确认新任务名称更新为合并后名称

---

## 删除确认规范

### 暂停任务删除

长期 PAUSED 的任务，按以下步骤处理：
1. 列出所有 PAUSED 任务
2. 标记超过 14 天的
3. 询问用户确认后删除

### 被替代任务删除

已被新任务覆盖的旧任务：
1. 确认新任务已覆盖旧任务全部功能
2. 直接删除旧任务（无需再次确认，因为功能已迁移）

## 完整审计流程

### Step 1：获取所有定时任务

列出所有任务：ID / 名称 / 状态 / 频率 / 工作目录

### Step 2：Token 消耗统计

读取数据库，分析上一个自然月每个任务的 token 消耗，按总消耗从高到低排序。

### Step 3：合并评估

遍历所有 ACTIVE 任务，按合并规则评估：

1. **按触发时间分组**：找出同一时间段（±30分钟）的任务
2. **按领域分组**：找出同领域的任务
3. **按功能覆盖检查**：找出功能被新任务覆盖的旧任务
4. **检查重复名称**：找出完全同名的任务

### Step 4：冗余清理评估

1. 列出所有 PAUSED 任务，标记超过 14 天的
2. 对比功能覆盖关系，标记被替代的任务
3. 评估每日执行任务的降频可行性

### Step 5：输出综合报告

```markdown
## 📊 定时任务综合审计报告（YYYY-MM）

### 📈 Token 消耗概览
- 自动化任务总数：X 个（ACTIVE: X / PAUSED: X）
- 总运行次数：X 次
- 总 token 消耗：XXX 万 tokens
- Top 5 高消耗任务

### 🔴 高消耗任务（需优化 prompt）
| 排名 | 任务名 | 月消耗 | 每次均值 | 优化方向 |

### 🔀 可合并任务
| 合并组 | 任务 A | 任务 B | 合并理由 | 建议操作 |

### 🗑️ 可删除任务
| 任务名 | 状态 | 删除理由 |

### 💡 降频建议
| 任务名 | 当前频率 | 建议频率 | 理由 |

### ✅ 综合优化方案
1. 🔀 合并 N 组任务 → 节省 N-1 次触发
2. 🗑️ 删除 N 个冗余任务 → 减少任务列表噪音
3. ⏰ 降频 N 个任务 → 节省 XX% token
4. ✂️ 精简 N 个 prompt → 节省 XX% token

### 📋 执行清单
- [ ] 合并组1：{任务A} + {任务B} → {新名称}
- [ ] 删除：{列表}
- [ ] 降频：{列表}
```

## 技术细节

### 数据库查询

本技能通过读取定时任务管理系统的数据库来统计 token 消耗。关键表结构：
- 会话表：`is_background_automation = 1` 标识定时任务触发的会话
- 使用量表：`input_tokens` / `output_tokens` 记录每次 token 消耗
- 时间字段为毫秒级时间戳

### 依赖

- 定时任务管理工具（列出/查看/更新/删除）
- 数据库读取能力（sqlite3 或等效工具）

---

## 注意事项

1. **合并前必须读取原任务完整内容**，确保理解全部功能后再合并
2. **PAUSED 任务删除需用户确认**，不可自动删除（除非已被替代）
3. **合并后及时更新任务内容**，避免残留旧引用
4. **数据库时间字段为毫秒级**，查询时需注意时间戳转换
